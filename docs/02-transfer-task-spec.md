# Этап 2. Next.js сайт бронирования поездок

Срок: 3-4 дня.

## 1. Контекст

На этапе 2 нужно собрать новый проект по отдельному клиентскому ТЗ: сайт транспортной компании с лендинговой страницей, продуктами, формой бронирования поездки и страницей моковых бронирований для менеджера.

Клиентский текст находится в файле 02-transfer-client-content.md.

Цель этапа - проверить работу с Next.js, React, TypeScript, Tailwind CSS, SEO, server/client components, мультиязычностью, формами и интерактивным UI.

## 2. Обязательный стек

- Next.js с App Router.

- React.

- TypeScript.

- Tailwind CSS v4.

- next-intl для мультиязычности.

- react-hook-form для обработки форм.

- zod или valibot для схемы валидации.

Не заменять обязательные инструменты без согласования с ментором.

## 3. Языки

Сайт должен поддерживать два языка:

- английский;

- русский.

Основная версия сайта - английская. Русская версия должна быть доступна через переключатель языка.

Требования:

- использовать next-intl;

- хранить переводы в отдельных message-файлах;

- не хардкодить интерфейсные тексты прямо в компонентах, если они должны переводиться;

- перевести основные страницы, форму, ошибки валидации и статусы бронирований;

- metadata для основных страниц должны соответствовать текущей локали.

## 4. Что нужно сделать

Собрать Next.js приложение, в котором есть:

- небольшая лендинговая страница;

- список нескольких типов поездок;

- форма бронирования поездки;

- страница со списком забронированных поездок для менеджера на предзаполненных mock-данных;

- визуальные состояния формы без сохранения отправленных данных;

- уместные анимации, которые улучшают восприятие интерфейса.

Страница менеджера пока без авторизации. Она нужна только для иллюстрации внутреннего сценария и подготовки к следующему этапу.

## 5. Страницы

В проекте должны быть страницы или route-сегменты для следующих сценариев:

- главная лендинговая страница;

- продукты или типы поездок;

- форма бронирования;

- список бронирований для менеджера;

- юридические страницы или разделы: privacy policy, cookie policy, terms.

## 6. Форма бронирования

Форма должна быть реализована через react-hook-form и zod или valibot.

Форма должна собирать:

- имя и фамилию;

- компанию, если применимо;

- телефон;

- email;

- тип услуги;

- адрес подачи;

- пункт назначения;

- дату поездки;

- время подачи;

- количество пассажиров;

- необходимость встречи в аэропорту;

- дополнительный комментарий;

- согласие на обработку данных.

Обязательные поля стажер определяет самостоятельно на основе клиентского текста и должен объяснить это решение на ревью.

Минимальная валидация:

- обязательные поля не пустые;

- email похож на email;

- количество пассажиров больше 0;

- дата поездки не пустая;

- согласие на обработку данных обязательно.

После успешной отправки форма должна:

- показать success state;

- визуально показать, что запрос принят в обработку;

- остаться на странице формы или очистить форму.

Важно: на этапе 2 отправка формы не должна сохранять бронирование в список, localStorage, mock-store или backend. Список бронирований для менеджера живет отдельно от формы.

## 7. Моковые бронирования

Backend на этом этапе не нужен. Страница бронирований должна работать на заранее подготовленных mock-данных.

Форма бронирования не добавляет новые элементы в этот список.

Структуру данных стажер проектирует самостоятельно. Она должна быть типизирована.

Страница бронирований должна позволять понять внутренний сценарий менеджера:

- список бронирований;

- ключевые данные поездки;

- статус;

- фильтр или поиск по mock-данным;

- empty state;

- визуально понятное представление записей.

Не нужно связывать страницу бронирований с формой до backend-этапа.

## 8. SEO и рендеринг

Нужно показать понимание Next.js:

- использовать App Router;

- задать metadata или generateMetadata для основных страниц;

- статический контент лендинга и legal-страниц рендерить через server components;

- интерактивные части формы, фильтров и переключателя языка вынести в client components;

- уметь объяснить, почему конкретный компонент является server или client component.

## 9. UI, дизайн и анимации

Готового макета нет. Стажер должен с помощью AI-инструментов разработать дизайн и сверстать страницы.

Требования:

- интерфейс выглядит как современный сайт транспортной услуги, а не как документ с текстом;

- есть понятная визуальная иерархия;

- продукты и преимущества визуально структурированы;

- форма удобна для заполнения;

- список бронирований читается как рабочая внутренняя страница;

- интерфейс адаптивен;

- анимации уместны и не мешают пользоваться сайтом.

Анимации могут быть реализованы через motion, css или иную легкую библиотеку по согласованию. Не тратить большую часть времени на сложные эффекты.

## 10. Требования к архитектуре

- Понятная структура проекта.

- Локали и переводы организованы предсказуемо.

- Типы данных описаны явно.

- Схема валидации формы вынесена отдельно от JSX.

- Моковые данные вынесены отдельно от компонентов.

- UI-компоненты не превращены в один большой файл.

- README содержит команды установки и запуска.

- Заполнены TASK.md, PROMPT_PLAN.md и AI_USAGE.md.

## 11. Что нужно объяснить на ревью

Будьте готовы объяснить:

- как устроены маршруты App Router;

- как подключен next-intl;

- где лежат переводы и как добавляется новый язык;

- где server components, а где client components;

- как работает форма и схема валидации;

- как устроены типы для формы, mock-бронирований и типов поездок;

- какие AI-запросы помогли разработать дизайн;

- что было доработано руками после AI.

## 12. Критерии приемки

Проект принимается, если:

- приложение запускается;

- используется Next.js App Router;

- используется TypeScript;

- используется Tailwind CSS v4;

- используется next-intl;

- есть английская и русская версии сайта;

- используется react-hook-form;

- используется zod или valibot;

- есть лендинговая страница;

- есть минимум 4 продукта или типа поездок;

- есть форма бронирования с валидацией;

- есть страница бронирований для менеджера на предзаполненных mock-данных;

- форма отображает состояние отправки/ошибки;

- backend не используется;

- есть SEO metadata для основных страниц;

- server/client components разделены осознанно;

- интерфейс адаптивен;

- есть уместные анимации;

- README объясняет запуск;

- заполнен AI_USAGE.md;

- стажер может объяснить архитектуру и state flow.
