# AGENTS.md

Инструкции для AI-агентов (Claude Code, OpenCode и др.), работающих в этом репозитории.
Если ты агент — прочти этот файл целиком перед первым изменением кода.

## 1. Что это за проект

Сайт бронирования трансферов для транспортной компании **LONDON ROUTE TRANSFERS**
(UK, Private Limited Company). Двуязычный сайт (en — основной, ru — через переключатель):
лендинг, страница услуг, форма бронирования (без сохранения данных), страница
бронирований для менеджера на моках, юридические страницы.

Источники правды по контенту и требованиям — файлы в `docs/`:
- краткое summary задачи (было `TASK.md`)
- полное ТЗ и критерии приёмки (было `task-spec.md`)
- исходный клиентский текст (было `client-content.md`)

Прочти их до начала работы, не додумывай контент и требования — они там уже есть.
Юридический текст (privacy, cookies, terms, transport-terms) берётся дословно по смыслу
из клиентского текста и переводится — не сочиняется агентом заново.

Не предлагай замену (например, react-i18next вместо next-intl, или Formik вместо

## 3. Жёсткие ограничения (backend отсутствует)

- Никакого backend, API-роутов с реальной логикой, БД, внешних сервисов, localStorage,
  реальных HTTP-вызовов.
- Форма бронирования **не сохраняет** данные никуда. После сабмита — success/error
  UI-состояние (можно с искусственной задержкой через setTimeout для реалистичности
  loading-state), форма очищается.
- Список бронирований менеджера — **отдельный от формы** статичный typed mock-массив.
  Форма не должна пушить туда новые записи. Это отдельное требование ревьюера —
  не путай эти два потока данных между собой.
- Страница менеджера — без авторизации на этом этапе, это нормально, не добавляй guard.

      manager/page.tsx        # список бронирований — client component

      transport-terms/page.tsx# server component
  components/
    ui/                       # переиспользуемые UI-примитивы
    sections/                 # секции лендинга (hero, about, services, advantages)
    booking-form/             # client components формы
    manager/                  # filters, booking-list, empty-state
    legal/                    # компоненты юридических страниц
    layout/                   # header, footer, language-switcher (client)
  i18n/
    routing.ts                # конфигурация next-intl routing

  messages/
    en.json
    ru.json
  types/

Не сваливай всё в один файл. UI-компоненты, схемы валидации и моки — разнесены
по разным модулям (это отдельный пункт критериев приёмки).

## 5. Server vs Client components

- **Server (по умолчанию)**: лендинг (hero, about, services, advantages), страница
  услуг, все legal-страницы, header, footer, `layout.tsx`, `generateMetadata`.
- **Client (`"use client"`)**: форма бронирования (react-hook-form, состояние,
  валидация), фильтр/поиск на странице менеджера, переключатель языка, cookie-баннер,
  любые анимации/интерактив.
- Компонент становится client только тогда, когда ему реально нужны хуки, события
  браузера или состояние. Не помечай `"use client"` "на всякий случай" — на ревью
  попросят объяснить каждый конкретный выбор, не просто "форма и фильтры на клиенте".

## 6. i18n (next-intl)

- Переводы — только в `messages/en.json` и `messages/ru.json`, никакого хардкода
  интерфейсных строк в компонентах — только через `useTranslations` / `getTranslations`.
- Ключи переводов — вложенный объект по секциям (`hero.title`, `booking.form.name`,
  `validation.email`, `manager.status.confirmed` и т.п.), не плоский список.
- Английская версия — основная и содержательно первична; русская — перевод
  клиентского текста, не наоборот.
- `generateMetadata` для каждой страницы, на каждой локали — title/description
  на языке текущей локали.
- Ошибки валидации формы и статусы бронирований тоже идут через переводы.
- Добавление нового языка: новый JSON в `messages/` + локаль в конфигурации
  next-intl routing (`i18n/routing.ts`).

## 7. Форма бронирования

Поля — см. клиентский текст в `docs/`. **Обязательность полей агент определяет
самостоятельно** на основе клиентского текста (там явно перечислено, что
"обязательно") — не бери список как готовый факт, реши осознанно и будь готов
объяснить решение на ревью.

Поля: имя и фамилия, компания (опционально), телефон, email, тип услуги,
адрес подачи, пункт назначения, дата поездки, время подачи, количество пассажиров,
встреча в аэропорту (да/нет), комментарий, согласие на обработку данных.

Минимальная валидация:
- обязательные поля не пустые;
- email — валидный формат;
- количество пассажиров > 0;
- дата поездки не пустая;
- чекбокс согласия — обязателен.

Схема валидации — в `lib/validation.ts`, отдельно от компонента формы.
States: `idle` → `submitting` → `success` / `error`. После success — визуальное
подтверждение, форма остаётся на странице или очищается — без записи данных
куда-либо (см. п.3).

## 8. Mock-бронирования для менеджера

Типизированная структура записи (`types/mock-bookings.ts`):
`id`, `clientName`, `company?`, `serviceType`, `pickup`, `destination`, `date`,
`time`, `passengers`, `status` (`pending` / `confirmed` / `completed` / `cancelled`),
`notes?`.

Страница: список, ключевые данные поездки, статус (визуально отличим — бейджи/цвета),
фильтр/поиск по mock-данным, empty state (когда фильтр ничего не находит).

## 9. Юридические страницы

Четыре страницы, контент — перевод соответствующих разделов клиентского текста
из `docs/` (не выдумывать юридический текст самостоятельно): Privacy Policy,
Cookie Policy, Terms of Use, Transport Services Terms. Все — server components.

## 10. Дизайн и анимации

Готового макета нет — дизайн придумывается по AI. Требования: выглядит как
современный сайт транспортной услуги (не как текстовый документ), понятная
визуальная иерархия, услуги/преимущества визуально структурированы, форма удобна,
список бронирований читается как рабочий внутренний экран, адаптивность,
уместные (не перегруженные) анимации через CSS transitions / framer-motion или
аналог — не тратить большую часть времени на сложные эффекты.

## 13. Чего агенту делать не нужно

- Не добавлять backend, API-роуты с реальной логикой, БД, localStorage.
- Не подключать сторонние UI-киты без необходимости (Tailwind v4 — основа стилизации).
- Не связывать форму и список бронирований.
- Не хардкодить тексты интерфейса вместо ключей перевода.
- Не менять обязательный стек.
- Не сочинять юридический текст самостоятельно — переводить из клиентского текста.
- Не помечать компонент `"use client"` без конкретной причины (хук/событие/состояние).
